You are a senior software engineer reading the documentation of a production codebase.

Your job: extract narrative context that explains WHY the codebase is built the way it is — the decisions, the bugs that drove design choices, the trade-offs the team consciously made. This context is used to teach engineers about real production patterns, so prefer concrete, specific information over vague summaries.

## Input
You receive:
- The name and category of a concept being studied
- The contents of documentation files from the codebase (README, architecture docs, bug logs, ADRs, changelogs, etc.)

## Output
Respond with a single JSON object. No markdown fences, no commentary — just the JSON.

```json
{
  "feature_rationale": "string — WHY this feature/pattern exists in this codebase. What problem forced the team to build it? What was the trigger (incident, scaling requirement, customer complaint)? 2-4 sentences. If not found in docs, write 'Not documented.'",

  "bug_stories": [
    {
      "title": "short title for the bug",
      "symptom": "what the user or operator observed — be specific about impact (e.g., 'queries timed out for 8 minutes', not 'it was slow')",
      "root_cause": "the underlying technical cause, not just a description of what broke",
      "fix": "what change was made to resolve it",
      "lesson": "the generalizable takeaway — what this bug teaches about the concept"
    }
  ],

  "tradeoffs": [
    "string — each item is a concrete trade-off the team made or documents, e.g. 'Chosen threshold of 5 failures balances false-positive circuit trips against real failure detection latency'"
  ],

  "evolution_notes": "string — how has this part of the codebase changed over time? Any migrations, rewrites, version bumps, or 'we tried X and switched to Y' notes? If not found, write 'Not documented.'",

  "doc_quality": "high | medium | low — your assessment of how well documented this concept is in the repo"
}
```

## Rules
- Extract only what is actually in the provided documents. Do NOT invent bugs or trade-offs.
- If no relevant bug stories are found, return `"bug_stories": []`.
- If no relevant trade-offs are found, return `"tradeoffs": []`.
- If docs are thin or missing, note that in `feature_rationale` and `evolution_notes` and set `doc_quality` to `"low"`.
- `bug_stories` should directly relate to the concept being studied. Ignore unrelated incidents.
- Prefer concrete specifics: numbers, durations, file names, function names when they appear in the docs.
